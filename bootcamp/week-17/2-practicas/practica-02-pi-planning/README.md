# Práctica 02 — PI Planning Simplificado

## Escenario: EduRed — Red de escuelas privadas

**EduRed** gestiona la plataforma digital de una red de 15 escuelas privadas.
El equipo creció a 4 equipos de desarrollo (28 personas) y adoptó SAFe hace 2 meses.

El Product Manager **Tomas** está preparando el primer PI Planning de EduRed.
La Scrum Master y próxima RTE **Luciana** lo ayuda a construir el PI Backlog.

**Equipos del ART:**
- **Equipo Alfa** — módulo de matrículas y pagos
- **Equipo Beta** — portal de padres (comunicación y notas)
- **Equipo Gama** — plataforma de clases online
- **Equipo Delta** — reportes y analítica para directivos

---

### Paso 1: Construir el Program Backlog

Tomas tiene estas iniciativas para el próximo PI (12 semanas):

| Iniciativa | Descripción |
| ---------- | ----------- |
| A | Los directivos necesitan ver el rendimiento académico por aula en tiempo real |
| B | Las familias quieren pagar la mensualidad sin ir a la escuela |
| C | Los profesores necesitan subir tareas y calificarlas en la plataforma |
| D | El equipo de TI necesita migrar la base de datos a la nube |

**Escribe una Feature SAFe para cada iniciativa** (formato: beneficio de negocio + criterio de aceptación).

---

### Paso 2: Asignar Features a equipos

> **Restricción de PI Planning**: cada equipo puede comprometer máximo 2 Features por PI.
> Los equipos deben declarar si hay dependencias entre ellos.

**Completa la tabla de Team PI Plans en `starter/plantilla.md`** — Paso 2.

---

### Paso 3: PI Objective del ART

Con los 4 Team PI Plans, **Luciana** consolida el PI Objective del ART.

**Formato esperado:**

> "Al final del PI [número], el ART de EduRed habrá entregado [X], [Y] y [Z],
> lo que permitirá a [usuario] [beneficio de negocio]."

**Abre `starter/plantilla.md`** y completa los 3 pasos.
