# 02 — Antipatrones en Eventos de Scrum

## Objetivos

- Reconocer antipatrones frecuentes en cada evento de Scrum
- Distinguir síntomas de causas raíz
- Proponer correcciones concretas para cada antipatrón

---

## 1. Antipatrones en el Sprint Planning

| Antipatrón | Síntoma | Corrección |
| ---------- | ------- | ---------- |
| **Sprint sin Goal** | El equipo toma ítems del Backlog sin un objetivo común | El PO formula el Sprint Goal antes del Planning |
| **Planning interminable** | La reunión dura 10+ horas y el equipo sale exhausto | Dividir en dos bloques: qué (1h) y cómo (restante) |
| **Estimación en Planning** | El equipo refina y estima en el Planning mismo | Llegar con ítems ya refinados del Backlog Refinement |

---

## 2. Antipatrones en el Daily Scrum

| Antipatrón | Síntoma | Corrección |
| ---------- | ------- | ---------- |
| **Stand-up de reporte** | Cada dev habla al SM, no al equipo | Reformular: "¿Qué hice para el Sprint Goal?" |
| **Daily de 45 minutos** | Se resuelven problemas técnicos en la reunion | Cortar: "Eso lo resolvemos en 5 min después" |
| **Daily sin Sprint Goal** | Nadie sabe si el trabajo avanza hacia el objetivo | Hacer visible el Sprint Goal durante el Daily |

---

## 3. Antipatrones en la Sprint Review

| Antipatrón | Síntoma | Corrección |
| ---------- | ------- | ---------- |
| **Demo sin audiencia** | Solo el equipo asiste, no hay stakeholders | El PO invita activamente a stakeholders relevantes |
| **Demo de código** | Se muestran pull requests, no valor al usuario | Demostrar flujos completos desde la perspectiva del usuario |
| **Review sin feedback** | El Product Backlog no cambia tras la Review | Preparar preguntas explícitas para los stakeholders |

---

## 4. Antipatrones en la Retrospectiva

| Antipatrón | Síntoma | Corrección |
| ---------- | ------- | ---------- |
| **Retro de quejas** | Se habla de problemas pero no hay acciones | Terminar con: "¿Qué haremos diferente la próxima semana?" |
| **Acciones sin dueño** | "Deberíamos mejorar la comunicación" | Cada acción: persona responsable + fecha límite |
| **Retro cancelada** | Se salta la Retro cuando hay presión de entrega | La Retro es el momento MÁS importante cuando hay presión |

---

## Checklist de Comprensión

- ¿Puedo identificar si un Daily Scrum es un reporte o una sincronización?
- ¿Sé qué preguntar en una Sprint Review para obtener feedback útil?
- ¿Entiendo por qué la Retrospectiva no puede tener acciones sin responsable?
- ¿Reconozco cuándo un Sprint Planning no tiene Sprint Goal?
