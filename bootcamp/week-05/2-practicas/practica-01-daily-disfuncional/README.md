# Práctica 01 — Daily Scrum Disfuncional

**Semana 05 — Eventos de Scrum**
**Tiempo estimado**: 1.5 horas

---

## Contexto

> **Escenario**: El equipo de desarrollo de **FlowTrack**, una plataforma
> de seguimiento de envíos, lleva 3 sprints usando Scrum. El Scrum Master,
> Rodrigo, nota que el Daily Scrum siempre dura 40 minutos y el equipo sale
> agotado. Los developers se quejan de que "no sirve para nada".

---

## Paso 1: Diagnosticar el Daily actual

En la siguiente transcripción de un Daily Scrum, identifica 4 antipatrones
distintos. Anótalos en la plantilla.

**Abre `starter/plantilla.md`** y completa la Sección 1.

```
[09:02] Rodrigo (SM): Empezamos. Ana, ¿qué hiciste ayer?
[09:03] Ana (Dev): Trabajé en el módulo de tracking. Tuve un bug con el parser
        de JSON que me llevó 3 horas. Creo que el problema está en la librería
        que elegimos. Debería cambiarse la librería por otra más estable.
[09:08] Rodrigo: Ok, tómalo con calma. Marcos, ¿tú?
[09:09] Marcos (Dev): Terminé la pantalla de login. Pero necesito que alguien
        revise el diseño porque Laura (diseñadora, ausente) dijo que no cumple
        los criterios pero no me explicó cuáles.
[09:14] Rodrigo: Pido a Laura que mande los criterios. Sofía, ¿qué tienes?
[09:15] Sofía (Dev): Yo solo quería actualizar que el servidor de staging sigue
        caído desde el viernes. TI dice que lo arreglan "cuando puedan".
[09:17] Rodrigo: Anotado. Voy a escalar con TI. Dicho eso, ¿alguien tiene temas
        del backlog que quiera revisar? Porque aprovechememos que estamos todos...
[09:44] [Fin del Daily]
```

---

## Paso 2: Reformular el Daily

Rodrigo quiere hacer un Daily Scrum correcto la próxima vez. Define:

- Las 3 preguntas correctas que cada Developer debería responder
- Cómo manejar el bug de Ana (impedimento técnico)
- Cómo manejar el server de staging caído (impedimento externo)

**Abre `starter/plantilla.md`** y completa la Sección 2.
