## 📋 Description

An intensive **24-week (~6-month)** bootcamp designed to take students from zero knowledge in agile project management to the level of **Junior Scrum Master**, **Junior Product Owner**, or **Junior Agile Coach**.

All bootcamp artifacts are **Markdown** files — User Stories, Sprint Backlogs, Definitions of Done, retrospectives, OKRs and more — built to be used and adapted directly in real teams. This repository is **open source (MIT)**: you can copy, adapt, and reuse the templates in your own projects.

### 🏛️ Unique Domain Policy (Anti-Plagiarism)

Each learner works on projects within an **exclusively assigned domain** (banking app, e-commerce, educational platform, etc.). This ensures original implementations and prevents copying between classmates.

### 🎯 Learning Objectives

By the end of the bootcamp, students will be able to:

- ✅ Apply the values and principles of the **Agile Manifesto** on real projects
- ✅ Facilitate all **Scrum** events: Planning, Daily, Review, Retrospective
- ✅ Write and prioritize **User Stories** with Gherkin acceptance criteria
- ✅ Build and refine an ordered and estimated **Product Backlog**
- ✅ Use agile metrics: **Velocity, Burndown Chart, CFD**
- ✅ Scale Scrum using frameworks like **SAFe, LeSS, or Nexus**
- ✅ Apply **Agile Coaching** and advanced facilitation techniques
- ✅ Prepare for certifications: **PSM I, PSPO I, or PMI-ACP**

### 🚀 Why Agile with GitHub Projects and Markdown?

The bootcamp uses tools that are already part of real teams' workflows:

| Tool | Role in the bootcamp |
|---|---|
| **GitHub Projects** | Simulation of real Scrum/Kanban boards |
| **Markdown** | Artifacts: User Stories, Sprint Goals, DoD, retros |
| **Git** | Version control of deliverables |
| **VS Code** | Main editor for all `.md` files |

## 📚 Weekly Content

Each week follows this standard structure:

```
bootcamp/week-XX/
├── README.md                  # Week description and objectives
├── rubrica-evaluacion.md      # Evaluation criteria (3 evidence types)
├── 0-assets/                  # SVG diagrams (flows, boards, maps)
├── 1-teoria/                  # Theory material in Markdown (~120 lines/file)
├── 2-practicas/               # Guided simulations with templates
│   └── practica-XX/
│       ├── README.md          # Step-by-step instructions
│       ├── starter/           # Template with sections to complete
│       └── solution/          # Commented reference solution
├── 3-proyecto/                # Weekly integrating project (assigned domain)
│   └── starter/
│       ├── contexto.md        # Adaptable generic scenario
│       └── entregables.md     # TODOs for the learner
├── 4-recursos/                # Free ebooks, videography, webography
└── 5-glosario/                # Key agile terms (A–Z)
```

### 🔑 Key Components

| Component | Description |
|---|---|
| **Theory** | Concise explanations (max 120 lines), examples and situational scenarios |
| **Practices** | Guided simulations with commented templates — the learner completes, not invents |
| **Project** | Artifact applied to the assigned domain (User Stories, Sprint, OKRs, etc.) |
| **Rubric** | 3 evidence types: Knowledge 30% · Performance 40% · Product 30% |

## 📊 Learning Methodology

The bootcamp uses these teaching strategies:

- **Project-Based Learning (PBL)**: weekly projects on real products
- **Unique Domains**: each learner uses their assigned product across all projects
- **Role Playing**: simulations of Scrum events (Planning, Daily, Review, Retro)
- **Case Studies**: analysis of teams that adopted (or failed to adopt) agile
- **Deliberate Practice**: exercises with incremental complexity week by week
- **Peer Review**: feedback between learners on artifacts

### Time distribution (8h/week)

| Activity | Hours |
|---|---|
| Theory | 2–2.5h |
| Guided practices | 3–3.5h |
| Integrating project | 2–2.5h |

### Evaluation

| Evidence type | Weight |
|---|---|
| 🧠 Knowledge (concepts, anti-patterns) | 30% |
| 💪 Performance (practical simulations) | 40% |
| 📦 Product (functional artifact) | 30% |

**Passing**: minimum 70% in each type · timely delivery · originality

## 📝 Open Source Templates

1. **Fork** the repository
